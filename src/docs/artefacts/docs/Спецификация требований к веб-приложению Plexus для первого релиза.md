
# Спецификация требований к веб-приложению Plexus для первого релиза
--- 
## Задача
---
Завершить спецификацию требований к ПО для первого релиза Plexus:
* Дополнить спецификацию функциональными требованиями для функции заметок для фиксирования мыслей и идей в базе знаний;
* Дополнить спецификацию функциональными требованиями для функции директорий для одного из возможных(1) видов структуризации заметок в базе знаний;
* Дополнить спецификацию функциональными требованиями для функции проектов для фиксирования конечного набора последовательностей решений, нацеленных на выполнение какого-либо результата.
* Описать требования к выше обозначенным функциям. Для этого необходимо:
	* Создать диаграмму UML Use Case;
	* Создать ER-диаграмму;
	* Добавить табличные варианты использования;
	* Написать функциональные требования;
* Написать разделы спецификации:
	* атрибутов качества;
	* бизнес-правил;
	* глоссария.
1 - В базе знаний заметки могут быть структурированы тремя не взаимоисключающими способами - директориями (строгая иерархия папок, хранящих заметки подобно файловой системе), категориями (уникальная заметка; заметки одновременно содержат 0 или 1 категорию) и тегами (уникальными идентификаторами в содержании самих заметок; заметки одновременно содержат 0, 1 или много тегов).

### 1. Требования
#### 0.1. Бизнес-требования первого релиза

#### 0.2 Архитектурные требования
##### 0.2.1 Архитектурные ограничения

| ID     | Требование                                                                                                                                                 |
| ------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| АРХ-01 | Веб-приложение строится по клиент-серверной модели; фронтенд — SPA (Single Page Application); бэкенд предоставляет REST API                                |
| АРХ-02 | Мультитенантная архитектура: данные пространств изолированы на уровне строк (row-level security) внутри единой реляционной СУБД                            |
| АРХ-03 | Аутентификация реализуется через JWT-токены с поддержкой refresh-токенов; OAuth 2.0 используется для SSO-провайдеров                                       |
| АРХ-04 | Хранилище файлов (иконки, импортируемые вложения) — объектное хранилище, совместимое с S3 API                                                              |
| АРХ-05 | Все внешние API-интеграции (импорт из Notion/Obsidian) изолированы в отдельном адаптерном слое; замена или добавление интеграции не требует изменений ядра |
##### 0.2.2 Требования к развёртыванию
| ID     | Требование                                                                                                          |
| ------ | ------------------------------------------------------------------------------------------------------------------- |
| РЗВ-01 | Предусмотрены три окружения: `dev` (разработка), `staging` (нагрузочное тестирование и приёмка), `prod` (продуктив) |
| РЗВ-02 | Деплой на `prod` выполняется только через CI/CD-пайплайн с обязательным прохождением smoke-тестов на `staging`      |
| РЗВ-03 | Конфигурация окружений хранится в переменных окружения; секреты не хранятся в репозитории                           |
| РЗВ-04 | Откат (rollback) любого релиза на `prod` выполняется не дольше 10 минут без ручного вмешательства в данные          |

##### 0.2.3 Эксплуатационные требования (SLA / RTO / RPO)
| ID     | Требование                                                                                                                                                                     |
| ------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| ОПЕ-01 | Целевая доступность сервиса (SLA): 99,5 % в месяц (не более ~3,6 ч плановых и внеплановых простоев)                                                                            |
| ОПЕ-02 | Целевое время восстановления (RTO): не более 1 часа с момента обнаружения критического сбоя до восстановления работоспособности                                                |
| ОПЕ-03 | Целевая точка восстановления (RPO): не более 15 минут потери данных при катастрофическом отказе (резервное копирование каждые 15 минут)                                        |
| ОПЕ-04 | Мониторинг: все backend-сервисы экспортируют метрики в формате, совместимом с Prometheus; алерты настроены на: доступность эндпоинтов, латентность p95 > 1 с, error rate > 1 % |
| ОПЕ-05 | Логирование: структурированные JSON-логи уровней ERROR/WARN/INFO; ротация и хранение логов — не менее 90 дней                                                                  |
| ОПЕ-06 | Резервное копирование: полный бэкап БД — ежедневно в 02:00 UTC, инкрементальный — каждые 15 минут; бэкапы хранятся 30 дней; процедура восстановления тестируется ежемесячно    |
##### 0.2.4 Интеграционные требования
| ID     | Требование                                                                                                                                                                   |
| ------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ИНТ-01 | Внешние системы первого релиза: SMTP-провайдер (email-уведомления и подтверждения), SMS-провайдер (OTP для номера телефона), OAuth-провайдеры (Google, VK, Telegram, Яндекс) |
| ИНТ-02 | Взаимодействие с каждым внешним провайдером изолировано в отдельном адаптере с таймаутом 5 с и retry-политикой: 3 попытки с экспоненциальной задержкой                       |
| ИНТ-03 | Публичный REST API для импорта поддерживает версионирование (`/api/v1/...`); устаревшая версия не отключается ранее чем через 6 месяцев после публикации новой               |
| ИНТ-04 | Для OAuth-авторизации поддерживается стандарт OpenID Connect; новый провайдер подключается без изменения бизнес-логики                                                       |

Ответы на вопросы рецензии и соответствующие правила:

1. Если за 20 минут пользователь сделал 100 правок — одна версия в конце интервала или каждое изменение? Фиксируется одна версия по истечении интервала, отражающая итоговое состояние объекта на момент истечения таймера (debounce-подход), а не каждое отдельное изменение.
2. Что фиксируется в snapshot — как закодированы теги/ссылки? Snapshot хранит содержимое в исходном Markdown-представлении как есть (теги и ссылки остаются текстовыми конструкциями #тег и [[ссылка]] внутри текста, отдельно не нормализуются).
3. Может ли Редактор восстановить версию? Нет — восстановление версии доступно только Владельцу и Администратору (см. таблицу ВИ-05); Редактор может только просматривать историю.
4. Версии создаются для содержимого или для метаданных тоже? Уже зафиксировано в БП-Ф-03 («содержимое и метаданные»). Уточняется: оба фиксируются в одном snapshot-объекте (единая запись версии с полями content_snapshot и metadata_snapshot).
5. Правило о хранении версий: бесконечно? До какого числа? После soft-delete объекта? См. новое правило БП-В-61 ниже.

##### 1.1.3 Общие функции (все модули)
* УправлятьПользователями: Plexus должно предоставлять возможность управления составом пользователей пространства
	* ПригласитьПользователя: Plexus должно предоставлять возможность приглашения пользователя в пространство с назначением роли 
	* УдалитьПользователя: Plexus должно предоставлять возможность удаления пользователя из пространства с отзывом всех его прав; при удалении пользователя из пространства все созданные им объекты (заметки, директории, проекты, решения) сохраняются в пространстве; поле автора обнуляется и отображается как «Удалённый пользователь»
	* ИзменитьРольПользователя: Plexus должно предоставлять возможность изменения роли существующего участника пространства; Plexus должно запрещать пользователю понижение собственной роли; Администратор не может понизить себя до Пользователя, Владелец не может отказаться от роли Владельца
* УправлятьПравамиДоступа: Plexus должно предоставлять возможность управления разрешениями для ролей и отдельных пользователей
	* ИзменитьПраваРоли: Plexus должно предоставлять возможность изменения набора разрешений для выбранной роли с применением изменений ко всем её носителям
	* ИзменитьПраваПользователя: Plexus должно предоставлять возможность переопределения прав конкретного пользователя поверх его роли
* УправлятьПространством: Plexus должно предоставлять возможность создания, редактирования и удаления пространства базы знаний
	* СоздатьПространство: Plexus должно предоставлять возможность создания нового пространства с назначением создавшего пользователя его Владельцем 
	* РедактироватьПространство: Plexus должно предоставлять возможность изменения названия, описания и настроек видимости пространства
	* УдалитьПространство: Plexus должно предоставлять возможность удаления пространства со всем его содержимым после явного подтверждения Владельцем 
* УправлятьАдминистраторами: Plexus должно предоставлять возможность управления составом администраторов пространства исключительно Владельцем 
	* НазначитьАдминистратора: Plexus должно предоставлять возможность назначения роли Администратора существующему участнику пространства 
	* СнятьАдминистратора: Plexus должно предоставлять возможность понижения роли Администратора до Пользователя
* УправлятьВерсиями: Plexus должно предоставлять возможность просмотра истории изменений объектов и восстановления их предыдущих версий 
	* НастроитьИнтервалАвтосохранения: Plexus должно предоставлять возможность изменения интервала автосохранения версий; значение по умолчанию — 20 минут
	* ПросмотретьИсториюВерсий: Plexus должно предоставлять возможность просмотра списка версий объекта с датой, временем и автором каждого изменения; версии фиксируют как содержимое объекта, так и его метаданные (название, теги, статус)
	* ВосстановитьВерсию: Plexus должно предоставлять возможность восстановления объекта к выбранной версии с сохранением текущего состояния в истории
* СменитьИконку: Plexus должно предоставлять возможность изменения иконки объекта
	* ВыбратьИконкуИзУстановленногоНабора: Plexus должно предоставлять возможность изменения иконки объекта из установленных Пользователем наборов иконок
	* УбратьИконкуОбъекта: Plexus должно предоставлять возможность убрать иконку объекта
* УправлятьНаборомИконок: Plexus должно предоставлять возможность управления набором иконок
	* СоздатьНаборИконок: Plexus должно предоставлять возможность создания наборов иконок
	* ИзменитьВидимостьНабора: Plexus должно предоставлять возможность изменения видимости набора иконок (`private`, `space`, `unlisted`, `public`)
	* УстановитьПубличныйНабор: Plexus должно предоставлять возможность установки публичных наборов иконок через "Галерею наборов"
	* УстановитьНаборПоСсылке: Plexus должно предоставлять возможность установки наборов со `unlisted` или `public`статусом по ссылке
	* УдалитьНаборПространства: Plexus должно предоставлять возможность удаления существующего набора пространства с каскадным сбросом всех иконок, примененных к объектам
	* ДобавитьИконкуВНабор: Plexus должно предоставлять возможность добавления иконки в набор иконок
		* ДобавитьИконкуЗагрузкойФайла: Plexus должно предоставлять возможность добавления иконки в набор иконок через загрузку файла
		* ДобавитьИконкуПоURL: Plexus должно предоставлять возможность добавления иконки в набор иконок через ввод URL-ссылки на изображение

##### 1.2.3 Функции модуля заметок
* СоздатьЗаметку: Plexus должно предоставлять возможность создания заметки
	* ДублироватьЗаметку: Plexus должно предоставлять возможность создания дублированием существующей заметки
	* СоздатьЗаметкуИзПустойСсылки: Plexus должно предоставлять возможность создания заметки переходом по висящей ссылке; висящая ссылка сохраняется бессрочно до момента создания заметки вручную или по переходу
* РедактироватьЗаметку: Plexus должно предоставлять возможность редактирования содержимого заметки и названия заметок
	* ДобавитьТег: тег фиксируется автоматическим считыванием выражения #(название_тега) из содержимого заметки; теги чувствительны к регистру (#Идея и #идея — разные теги); тег автоматически удаляется из пространства, когда перестаёт встречаться в любой заметке 
	* ДобавитьСсылку: Plexus должно предоставлять возможность редактирования содержимого, позволяя оставлять ссылки
		* ДобавитьСсылкуНаЗапись: Plexus должно предоставлять возможность редактирования содержимого, позволяя оставлять ссылки на заметку в пространстве; заметка может ссылаться на саму себя (в данном случае эта ссылка поместится только в forwardlinks); backlinks мягко удалённых заметок скрываются из списка ссылающихся; backlinks и forwardlinks обновляются автоматически
		* ДобавитьСсылкуНаРесурс: Plexus должно предоставлять возможность редактирования содержимого, позволяя оставлять ссылки на локальный или интернет-ресурс
* УдалитьЗаметку: Plexus должно предоставлять возможность удаления заметок
	* УдалитьМягкоЗаметку: Plexus должно предоставлять возможность удаления заметку с возможностью их восстановления через Корзину; мягко удалённые заметки сохраняют всё содержимое, включая комментарии и ссылки; редактирование и создание заметок в Корзине недоступно; срок хранения по умолчанию — 60 дней, Владелец вправе изменить
	* УдалитьПерманентноЗаметку: Plexus должно предоставлять возможность удаления из Корзины
	* ВосстановитьЗаметку: Plexus должно предоставлять возможность восстановления заметки из Корзины
* КомментироватьЗаметку: Plexus должно предоставлять возможность оставлять якорные комментариями на выделенных Пользователем фрагментах содержимого объекта или оставлять комментарий к целому объекту; комментарии сохраняются при мягком удалении и восстанавливаются вместе с заметкой
* ИмпортироватьЗаметку: Plexus должно предоставлять возможность импорта заметок
* НайтиЗаметку: Plexus должно предоставлять возможность поиска заметок
	* НайтиЗаметкуПоНазванию: Plexus должно предоставлять возможность поиска заметок по их названию с применением фильтров на ссылки и теги, по цельным словам и регистру, а также с использованием регулярных выражений
	* ГлубокийПоиск: Plexus должно предоставлять возможность поиска заметок по их названию и содержанию с применением фильтров на ссылки и теги, по цельным словам и регистру, а также с использованием регулярных выражений
* УправлятьШаблонами: шаблон — обычная заметка с флагом «шаблон»; все CRUD-операции для шаблонов идентичны операциям с заметками; изменение шаблона не затрагивает заметки, ранее созданные на его основе

##### 1.3.3. Функции модуля директорий
* СоздатьДиректорию: Plexus должно предоставлять возможность создания директории
  для структурирования заметок в базе знаний
    * ДублироватьДиректорию: Plexus должно предоставлять возможность создания директории дублированием существующей со всем её содержимым; дублирование глубокое — дублируются директория и всё её содержимое рекурсивно; имена дублей получают приписку «(n)», где n — порядковый номер копии начиная с 1 

* ИмпортироватьДиректории: Plexus должно предоставлять возможность импорта
  директорий с сохранением исходной иерархии
    * ИмпортироватьЧерезAPI: Plexus должно предоставлять возможность импорта
      директорий через API внешней базы знаний
    * ИмпортироватьФайлами: Plexus должно предоставлять возможность импорта
      директорий загрузкой файлов напрямую в пространство

* НайтиДиректорию: Plexus должно предоставлять возможность поиска директорий
    * НайтиДиректориюПоНазванию: Plexus должно предоставлять возможность поиска
      директорий по их названию с применением фильтров по цельным словам и регистру,
      а также с использованием регулярных выражений
    * ГлубокийПоискДиректорий: Plexus должно предоставлять возможность поиска
      директорий по их названию и содержимому вложенных заметок с теми же фильтрами

* СменитьИконкуДиректории: Plexus должно предоставлять возможность смены иконки
  директории для её визуальной идентификации
    * СменитьИконкуИзХранилища: Plexus должно предоставлять возможность установки
      иконки директории загрузкой изображения с локального устройства пользователя
    * СменитьИконкуПоURL: Plexus должно предоставлять возможность установки иконки
      директории указанием URL-ссылки на изображение

* УправлятьДиректориями: Plexus должно предоставлять возможность административного
  управления директориями пространства
    * ОграничитьДоступКДиректории: Plexus должно предоставлять возможность
      ограничения доступа к директории и всему её содержимому для выбранных
      пользователей или ролей; права доступа заметки имеют приоритет над правами родительской директории — заметка с более широким доступом остаётся доступной даже внутри ограниченной директории
    * УдалитьДиректориюМягко: Plexus должно предоставлять возможность удаления директории с возможностью её восстановления через Корзину; при удалении непустой директории система явно предупреждает пользователя о наличии содержимого; все вложенные заметки удаляются вместе с директорией; глубина вложенности директорий не ограничена
    * УдалитьДиректориюПерманентно: Plexus должно предоставлять возможность
      безвозвратного удаления директории из Корзины
    * ВосстановитьДиректорию: Plexus должно предоставлять возможность восстановления директории из Корзины; при восстановлении директории из Корзины восстанавливаются все вложенные заметки, включая те, что были удалены отдельно

##### 1.4.3 Функции модуля проектов
* СоздатьПроект: Plexus должно предоставлять возможность создания проекта для фиксирования конечного набора последовательностей решений, нацеленных на достижение определённого результата
    * СоздатьПроектИзЗаметки: Plexus должно предоставлять возможность создания проекта из существующей заметки, используя её содержимое как описание проекта; связь двусторонняя — в свойствах заметки автоматически фиксируется, что из неё был создан проект; удаление или редактирование заметки не влияет на производный проект

* СоздатьРешение: Plexus должно предоставлять возможность создания решения в рамках существующего проекта
    * СоздатьРешениеИзЗаметки: Plexus должно предоставлять возможность создания решения из существующей заметки, используя её содержимое как описание решения; связь двусторонняя — в свойствах заметки автоматически фиксируется, что из неё было создано решение; удаление или редактирование заметки не влияет на производное решение  
    * СоздатьДочернееРешение: Plexus должно предоставлять возможность создания дочернего решения, подчинённого существующему решению того же проекта; глубина дерева решений не ограничена

* НайтиПроект: Plexus должно предоставлять возможность поиска проектов
    * НайтиПроектПоНазванию: Plexus должно предоставлять возможность поиска проектов по их названию с применением фильтров по цельным словам и регистру, а также с использованием регулярных выражений
    * ГлубокийПоискПроектов: Plexus должно предоставлять возможность поиска проектов по их названию и содержимому с теми же фильтрами

* НайтиРешение: Plexus должно предоставлять возможность поиска решений
    * НайтиРешениеПоНазванию: Plexus должно предоставлять возможность поиска
      решений по их названию с применением фильтров по цельным словам и регистру, а также с использованием регулярных выражений
    * ГлубокийПоискРешений: Plexus должно предоставлять возможность поиска решений по их названию и содержимому с теми же фильтрами
    * НайтиРешениеВПроекте: Plexus должно предоставлять возможность ограничения поиска решений рамками конкретного проекта

* УправлятьПроектами: Plexus должно предоставлять возможность административного
  управления проектами пространства
    * ОграничитьДоступКПроекту: Plexus должно предоставлять возможность ограничения доступа к проекту и всем его решениям для выбранных пользователей или ролей
    * УдалитьПроектМягко: Plexus должно предоставлять возможность удаления проекта с возможностью его восстановления через Корзину
    * УдалитьПроектПерманентно: Plexus должно предоставлять возможность безвозвратного удаления проекта из Корзины
    * ВосстановитьПроект: Plexus должно предоставлять возможность восстановления проекта из Корзины

* УправлятьРешениями: Plexus должно предоставлять возможность административного управления решениями в рамках проектов пространства
    * ПереместитьРешение: Plexus должно предоставлять возможность перемещения решения из одного проекта в другой
    * УдалитьРешениеМягко: Plexus должно предоставлять возможность удаления решения с возможностью его восстановления через Корзину
    * УдалитьРешениеПерманентно: Plexus должно предоставлять возможность безвозвратного удаления решения из Корзины
    * ВосстановитьРешение: Plexus должно предоставлять возможность восстановления решения из Корзины
* РедактироватьРешение: Plexus должно предоставлять возможность изменения названия и содержимого решений, находящихся в статусе «На рассмотрении»
* ПринятьРешение: Plexus должно предоставлять возможность перевода решения в статус «Принято»; право принятия решений имеют Владелец, Администратор и настраиваемые роли с явно выданным разрешением
    * ОтклонитьКонкурирующиеРешения: при принятии решения все прочие решения того же уровня и все их дочерние решения автоматически переводятся в статус «Неявно отклонено»; дочерние решения принятого решения остаются в статусе «На рассмотрении»
* ОтклонитьРешение: Plexus должно предоставлять возможность перевода решения в статус «Отклонено»; право отклонения решений имеют Владелец, Администратор и настраиваемые роли с явно выданным разрешением
    * ОтклонитьДочерниеРешения: При отклонении решения Plexus должно автоматически переводить все его дочерние решения в статус «Отклонено»; при явном отклонении решения все его дочерние решения автоматически переводятся в статус «Неявно отклонено»
* АннулироватьРешение: перевод принятого или отклонённого решения обратно в статус «На рассмотрении»; при аннулировании принятого решения все решения, переведённые в статус «Неявно отклонено» при его принятии, автоматически возвращаются в статус «На рассмотрении»; явно отклонённые решения сохраняют свой статус
* ПросмотретьДеревоРешений: Plexus должно предоставлять возможность просмотра полной иерархии решений проекта в виде дерева
    * ПерейтиКРодителю: Plexus должно предоставлять возможность навигации к родительскому решению из текущего узла дерева
    * ПерейтиКДочернимРешениям: Plexus должно предоставлять возможность
      навигации к дочерним решениям из текущего узла дерева
* ЗавершитьПроект: проект завершается автоматически при достижении листовых узлов всех деревьев решений (проект рассматривается как лес из одного или нескольких деревьев); проект может быть завершён вручную Владельцем или Администратором в любой момент

### 2. Атрибуты качества, бизнес-правила и глоссарий
_Статическое_ — правило является инвариантом системы: не зависит от текущего состояния данных, не может быть переопределено конфигурацией, истинно всегда. 
_Динамическое_ — правило зависит от состояния данных, конфигурации или действий пользователя; может давать разные результаты в разные моменты времени.
